# 03 — 判斷力外化矩陣（換路徑／完成定義／熔斷）

> 目的：把高階模型的直覺量化成弱模型可肉眼逐條比對的檢核表。每條附【完美正例】與【典型反例】。
> 使用方式：不用背。撞到對應情境時翻到對應區塊，逐條比對，照結論執行。
> 治理等級：🔒 鎖定檔（`05_maintenance.md`）。

## R1 — 換路徑信號：「方向錯了」長什麼樣子

出現**任一**信號 → 停止在原方案上重試，回到問題定義重新選路（換路徑後在 worklog「絕不重做」區記錄被放棄的方案與原因）：

| # | 信號 | 完美正例（正確反應） | 典型反例（弱模型常見錯誤） |
|---|---|---|---|
| 1 | 同一個錯誤訊息在兩次「不同的修法」後仍出現 | 停手。把兩次修法＋錯誤原文寫成錯誤軌跡，重新讀錯誤的**第一行與最後一行**，懷疑自己對錯誤的理解本身是錯的；必要時升級（02 §3） | 第三次、第四次繼續在同一行代碼上換寫法，每次都「感覺這次會過」 |
| 2 | 修一個「小 bug」的 diff 滾到 >3 個檔案 | 停手。小 bug 大 diff = 你在修症狀不是病因。回頭找單一根因；若根因真的橫跨多檔，先在 worklog 寫出因果鏈再動手 | 繼續加補丁，每個檔案都「順便」改一點，diff 越滾越大 |
| 3 | 為了讓測試變綠，你正在**修改測試**而不是實作 | 停手。先回答：「這條測試鎖的行為，規格上真的變了嗎？」有 issue/文件佐證才能動測試，且受保護 spec 需 P0-OVERRIDE | 把斷言改鬆、把 case 註解掉、給測試加 skip——gate 過了，bug 上線了 |
| 4 | 你發現「必須改凍結區檔案」才能完成任務 | 停手。99% 代表選錯了實作位置——新功能該落 `app/api/v2/**`。重讀任務，找不凍結的路徑；真是 P0 才走 override 協議 | 直接動手改 legacy route，被 file-guard 擋後開始想辦法繞 hook |
| 5 | 你在「猜」規格：找不到 AC、文件、或既有行為佐證，開始自行腦補欄位名／狀態值／文案 | 停手。先 Grep 既有代碼找同類先例；再查 `docs/`；都沒有 → 熔斷提問（R3） | 編一個看起來合理的欄位名寫進 DB gateway，下游全部靜默錯位 |
| 6 | 工具連續報錯 ≥2 次且錯誤訊息**變都沒變** | 停手。錯誤不變 = 你的修正根本沒被執行到（改錯檔？沒存檔？cache？）。先驗證修正真的生效（read-back），再談修錯 | 繼續改代碼，其實每次跑的都是同一份舊檔 |

## R2 — 完成定義（Definition of Done）：全部打勾才准說「完成」

宣稱任務完成／關 issue／請求 merge 前，逐條檢查。**任何一條不滿足，任務狀態只能寫「進行中」或「HOLD」，不得寫「完成」。**

- [ ] **D1 落檔證明**：宣稱建立/修改的每個檔案，都用 Read 或 `git show HEAD --stat` 重新確認存在且內容正確（防假性完成：「我已寫入」必須有 read-back 證據）。
- [ ] **D2 targeted 綠燈**：`.claude/hooks/run-checks.sh <本任務測試檔>` exit 0，證據在 `.claude/state/last-checks.json`。
- [ ] **D3 全套回歸**：開 PR 前 `run-checks.sh --all` 綠燈（或 CI 綠燈替代）。
- [ ] **D4 新行為有新測試**：backend → `tests/**/issueNNNN-*.test.mjs`；使用者可見 → `e2e/issueNNNN-*.spec.ts`（詳 `07_testing_playbook.md`）。
- [ ] **D5 已 commit＋push**：`git status` 乾淨、`git log origin/<branch>..HEAD` 為空。
- [ ] **D6 CI 綠燈才 merge**：check-run conclusion=success 的連結已記入 worklog。
- [ ] **D7 記憶錨點已雙寫**：worklog 更新＋issue 留言。
- [ ] **D8 AC 逐條對照**：issue 的每條 AC 有 PASS 證據或明確標註（`NOT_VERIFIED-live`＋blocker）。

【完美正例】「AC 3/3 PASS：D1 read-back ✓、D2 `run-checks.sh tests/api/issue1600-refund.test.mjs` exit 0（14:32）、D3 --all 綠、PR #1601 CI success（連結）、worklog+issue 已更新。」
【典型反例】「已完成修改並通過測試 ✅」——沒有檔名、沒有 exit code、沒有連結。**這句話在本專案等於「什麼都沒做」。**

## R3 — 熔斷觸發：何時必須停下來問使用者

出現**任一**條件 → 停止自主作業，整理現狀後提問（提問格式在下方）：

| # | 條件 | 說明 |
|---|---|---|
| 1 | 動作不可逆且觸及真實世界 | 會寄真實 email/LINE 通知、動到正式付款/退款、刪除生產資料、對外發布。冷啟動期間資料庫還乾淨，**不代表可以亂動**——習慣是現在養成的 |
| 2 | AC 有兩種以上合理解讀 | 各做一半的成本 > 問一次的成本。帶著「解讀 A vs 解讀 B ＋各自後果」去問 |
| 3 | 完成任務「必須」違反十條鐵律任一條 | 包括需要 P0-OVERRIDE 的情況——override 本來就只有使用者能給 |
| 4 | 同一問題 Opus 級也修不動兩輪 | 附完整錯誤軌跡（02 §3） |
| 5 | 品味／商業美感／品牌決策 | 文案語氣、視覺取捨、定價呈現、命名對外用詞。先查 `BRAND_BOOK.md`，查得到照辦；查不到→**這是弱模型的能力極限（01 §5.6），注定不可靠，不得自由發揮**。準備 2–3 個選項＋trade-off 再問 |
| 6 | 預估 diff >300 行或橫跨 >2 個 auth realm | 大手術先交計畫再動刀 |

**熔斷提問格式**（讓使用者 30 秒能回）：①一句話說卡在哪 ②我已試過/查過什麼 ③選項 A/B（各一句 trade-off）④我推薦哪個、為什麼。**禁止**只丟「請問怎麼辦」。

【完美正例】「退款文案 BRAND_BOOK 沒有覆蓋。選項 A『退款將於 7 個工作天內原路退回』（明確、保守）；B『我們已為您安排退款…』（品牌語氣、但承諾模糊）。推薦 A：付款相關文案準確性優先。」
【典型反例】自己寫了一句自認溫暖的退款文案直接上線；或反過來，每做三步就問一次「這樣可以嗎？」把使用者當 code reviewer。

## R4 — 誠實條款的行為版

1. 不確定的事：先查（Grep/docs/issue 歷史）→ 查不到標 `NOT_VERIFIED` ＋寫明查過哪裡 → **絕不編造**（欄位名、API 行為、數字、連結都算）。
2. 報告失敗跟報告成功用同樣的音量：測試紅了就貼紅的輸出，不粉飾、不省略。
3. 「我做了 X」永遠附證據指標（路徑、SHA、exit code、連結）。寫不出證據指標的句子，改寫成「我嘗試了 X，尚未驗證」。
