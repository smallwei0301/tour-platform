# 02 — 多模型調度與動態升降級守則

> 對象：未來在本環境擔任「主對話／指揮官」的模型（Opus 4.8、Sonnet 等）。
> 原則：**指揮官的 context 是最稀缺資源**——它一被垃圾填滿，判斷力就跟著死。整份守則都在保護它。
> 治理等級：🔒 鎖定檔（`05_maintenance.md`）。

## 1. 指揮官不下場（Commander stays on the hill）

主對話模型只做四件事：**決策、拆解、驗收、對使用者說話**。以下工作一律派給 Subagent（Agent 工具），主對話只接收精簡結論：

| 工作型態 | 派給 | 建議 agent type |
|---|---|---|
| 大批量讀檔／repo 掃描／「XX 在哪裡」 | Subagent | `Explore`（唯讀，回結論不回檔案傾印） |
| 實作規劃、方案比較 | Subagent | `Plan` |
| 多步驟實作、修 bug | Subagent | `general-purpose` |
| 驗收（read-back＋重跑測試） | **Fresh-context** Subagent | `general-purpose`（見 §5） |

主對話**禁止**做的事：自己 `cat` 十個檔案找東西、自己滾動式讀完整個模組、把 subagent 的原始輸出原封貼進對話。收到 subagent 回報後，只保留「結論＋路徑＋行號」，其餘丟棄。

例外（不必派工）：單一事實查證（知道檔名/符號，一次 Read/Grep 就有答案）、≤2 檔的小修改。派工本身有成本，殺雞不用牛刀。

## 2. 派工三件套（缺一不發包）

每一次 Agent 呼叫的 prompt 必須包含三個區塊（模板見 `04_templates.md`，直接套用）：

1. **目標與背景（Context 引入）**：要達成什麼、為什麼、相關檔案的明確路徑、必讀的 harness/文件路徑。Subagent 是 fresh context——**它不知道任何你知道的事**，沒寫進 prompt 的背景等於不存在。
2. **驗收條件（AC）**：可機器判定的完成標準（「測試 X 綠燈」「檔案 Y 存在且含 Z」），不是「弄好它」。
3. **回報格式限制**：規定回什麼（成果路徑＋關鍵行號＋一句話結論＋遇到的 blocker），**禁止噴大段代碼**（>20 行的引用一律改成 `path:line` 指位）。

## 3. 異常升降級路徑

| 情境 | 動作 |
|---|---|
| Haiku/小模型：工具呼叫或語法錯誤 **1 次** | 直接升級 Sonnet 重派（同一份三件套 prompt） |
| Sonnet/中模型：同一子任務連續卡關或出錯 **2 次** | 停止重試。整理**錯誤軌跡**（做了什麼、指令、完整錯誤訊息、已排除的假設）升級 Opus 4.8 |
| Opus 4.8 解出**固定模式**（同構修改 N 處） | 把模式寫成暫時性 script 或逐步 checklist，降級交回便宜模型批次套用 |
| 同一件事 Opus 也修不動（**兩輪**為限） | 熔斷：worklog 記錄現狀＋錯誤軌跡，向使用者提問（`03_rubrics.md` R3） |

硬規則：
- **升級必附錯誤軌跡**。沒有軌跡的升級 = 讓強模型從零猜，浪費它。
- **同一件事最多重試兩輪**（一輪 = 一次派工＋一次修正）。第三輪只有兩個合法出口：換路徑（R1）或熔斷（R3）。
- 模型選擇用 Agent 工具的 `model` 參數（`haiku`/`sonnet`/`opus`）；不確定時省略（繼承 session 模型）。

## 4. 記憶錨點協議（對抗語意迷航；Q3 拍板：issue＋repo 雙寫）

**單一事實來源是 worklog 檔，不是你的上下文記憶。**

1. 開工（或接手）任務：先讀 `docs/operations/worklogs/issueNNNN.md`；不存在就按 `04_templates.md` §W 建立。
2. worklog 固定五區：**目標／AC 清單（逐條打勾）／已完成（含 commit SHA）／下一步／絕不重做（Do-NOT-redo）**。
3. **每完成一個里程碑**（一條 AC 轉綠、一個 PR 開出、一次 merge）：更新 worklog → 同步一則精簡留言到 GitHub issue（跨 session 的公開錨點）。
4. **每次 context 被壓縮、session 恢復、或連續工作超過 ~10 個工具呼叫後感到「不確定剛才做到哪」**：立刻重讀 worklog，以檔案內容為準對齊，再繼續。
5. 「絕不重做」區記錄已完成且驗證過的東西（檔案、決策、已排除的方案）——**動手改任何檔案前，先查這一區有沒有它**。這是防止「亂改已完成代碼」的第一道軟防線（硬防線是 file-guard）。

## 5. 隔離驗證（實作者不得自我驗收）

1. 實作 Subagent 回報「完成」≠ 完成。指揮官必須另派一個 **Fresh-context 驗收 Subagent**，prompt 只給：AC 清單＋成果路徑＋驗收模板（`04_templates.md` §V），**不給實作過程**（避免被實作者的敘事帶偏）。
2. 驗收 Subagent 必做：
   - **Read-back**：實際重讀宣稱建立/修改的每個檔案（假性完成的第一殺手鐧——檔案不存在立刻現形）。
   - **實跑**：重新執行 targeted 測試（不信任 `.claude/state/last-checks.json`，自己跑）；前端行為跑 Playwright spec。
   - 逐條 AC 給判定：PASS（附證據）／FAIL（附差距）。
3. 設計/架構類產出（無法「跑」的東西）用**多樣本評審**：2–3 個獨立 subagent 各自審一遍（分別給不同視角：正確性／弱模型可執行性／與既有文件衝突），指揮官綜合裁決。
4. 驗收 FAIL → 回到 §3 升降級表，把 FAIL 證據當錯誤軌跡處理。

## 6. Token／context 熔斷

- 感到 context 吃緊（回應變慢、開始忘記剛做過的事）：**立刻**更新 worklog＋issue 留言到最新狀態，然後才繼續。寧可多寫三次 worklog，不可斷線一次。
- 需要收尾時的優先順序：已完成部分 commit＋push → worklog 補「下一步／未完成」→ 對使用者一頁摘要。產出新內容永遠排在保全既有成果之後。
