# Admin Operations Tracking UI Spec

> 目的：把營運追蹤模組從「欄位需求」推進到「Tracy 可直接設計與開發」的 UI 規格。
> 更新日期：2026-03-27

---

## 1. 模組定位

營運追蹤不是財務報表，而是 **MVP 階段的營運決策面板**。

這個模組要幫 Admin / CEO 快速回答：
- 這筆訂單有沒有賺
- 這筆訂單麻不麻煩
- 問題最多來自哪裡
- 哪個流程最值得優先自動化

---

## 2. 路由規劃

### 列表頁
- `/admin/operations-tracking`

### 詳情頁
- `/admin/operations-tracking/[orderId]`

---

## 3. 列表頁 UI 規格

### 3.1 頁面目標
讓 Admin 一眼看到：
- 哪些訂單健康
- 哪些訂單有退款 / 客訴 / 例外
- 哪些訂單單筆貢獻偏低

### 3.2 頁面區塊

#### A. 頁面標題列
- 標題：`營運追蹤`
- 說明文字：`追蹤每筆訂單的營運成本、例外事件與單筆貢獻`
- 操作按鈕：
  - `匯出 CSV`
  - `新增 / 編輯欄位`（可先不做）

#### B. Summary Cards（頂部摘要卡）
至少顯示 8 張卡：
1. 總 GMV
2. 平台總收入
3. 平均每單抽成收入
4. 平均每單人工時間
5. 平均每單人工成本
6. 退款率
7. 例外事件率
8. 平均單筆最終貢獻

#### C. Filter Bar（篩選區）
欄位：
- 日期區間
- 導遊
- 活動
- 是否退款
- 是否有例外事件
- 是否健康訂單
- 關鍵字搜尋（訂單編號 / 備註）

#### D. Data Table（主表格）
建議欄位順序：
- 訂單日期
- 訂單編號
- 導遊名稱
- 活動名稱
- 場次日期
- 旅客人數
- GMV
- 抽成收入
- 金流費
- 人工時間
- 人工成本
- 退款金額
- 例外事件標記
- 單筆最終貢獻
- 健康狀態
- 操作（查看詳情）

---

## 4. 表格欄位顯示規則

### 4.1 例外事件標記
使用 badge 顯示：
- `退款`
- `客訴`
- `改期`
- `導遊調整`
- `超賣 / 場次異常`

### 4.2 健康狀態
使用顏色標示：
- 綠色：健康訂單
- 黃色：可疑 / 需關注
- 紅色：不健康 / 虧損 / 多重例外

### 4.3 單筆最終貢獻
- 正數：綠色
- 低於門檻：橘色
- 負數：紅色

---

## 5. 詳情頁 UI 規格

### 5.1 頁面目標
讓 Admin 能對單一訂單進行完整回顧與編輯。

### 5.2 頁面區塊

#### A. Header
- 訂單編號
- 健康狀態 badge
- 返回列表按鈕

#### B. 基本資訊卡
顯示：
- 訂單日期
- 導遊名稱
- 活動名稱
- 場次日期
- 旅客人數
- GMV
- 平台抽成收入
- 金流費

#### C. 營運輸入區（Admin 可編輯）
表單欄位：
- 人工處理時間（分鐘）
- 人工成本估算
- 退款金額
- 補貼 / 補償金額
- 是否改期
- 是否退款
- 是否客訴
- 是否導遊臨時調整
- 是否超賣 / 場次異常
- 備註

#### D. 系統計算區
顯示：
- 單筆最終貢獻
- 健康狀態
- 最後更新時間

#### E. 關聯資訊區
顯示連結：
- 查看原訂單
- 查看退款紀錄
- 查看導遊資料
- 查看場次資料

---

## 6. 互動狀態

### 列表頁
- loading：Skeleton table
- empty：`目前沒有營運追蹤資料`
- filter active：顯示清除篩選按鈕

### 詳情頁
- save 成功：Toast `已更新營運追蹤資料`
- save 失敗：Toast `更新失敗，請稍後再試`
- 未填關鍵欄位：顯示提示，但 MVP 不強制阻擋

---

## 7. 行動裝置 / 響應式策略

### Desktop
- 完整 table + summary cards

### Tablet
- summary cards 2 欄排列
- table 橫向捲動

### Mobile
MVP 可降級為：
- 不主打 mobile admin
- 以詳情頁可看、可編輯為主
- 列表頁允許簡化顯示欄位

## 8. Tracy 開發優先順序

### P0
- 列表頁
- summary cards
- 詳情頁基本資訊
- Admin 可編輯欄位
- 最終貢獻計算顯示

### P1
- CSV 匯出
- Badge / 健康狀態顏色優化
- 篩選條件完整化

### P2
- 視覺優化
- 趨勢圖表
- 自訂欄位能力

---

## 9. 視覺原則

這個頁面不追求花哨，而追求：
- 一眼看出問題
- 一眼看出哪筆訂單要處理
- 一眼看出哪筆訂單在虧錢

所以視覺上要偏：
- 清楚的數字階層
- 明確的狀態顏色
- 資訊密度高，但不混亂

---

## 10. 核心原則

**營運追蹤 UI 的價值，不是報表好看，而是讓 CEO 與 Admin 有能力用真實訂單資料調整平台策略。**
