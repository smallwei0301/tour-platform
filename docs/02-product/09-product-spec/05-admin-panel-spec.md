# 後台管理系統規格（MVP 版）

> 目標：不是做豪華 ERP，而是讓平台在 MVP 階段能夠真實營運、人工介入、追蹤每筆訂單的健康度。
> 更新日期：2026-03-27

---

## 1. 後台角色（MVP）

MVP 階段先不做複雜 RBAC，僅保留一種 Admin 角色。

### Admin 能力
- 審核導遊申請
- 查看 / 搜尋所有訂單
- 查看 / 處理退款申請
- 查看 / 調整活動場次狀態
- 查看營運追蹤資料
- 手動補充營運事件與備註

---

## 2. 後台主模組

### A. 導遊審核
用途：處理導遊申請與 KYC。

功能：
- 申請列表
- 申請詳情
- 核准 / 退件 / 暫停
- 查看文件
- admin 備註

### B. 訂單管理
用途：追蹤所有訂單與例外情況。

功能：
- 訂單列表
- 篩選（日期 / 導遊 / 活動 / 狀態）
- 訂單詳情
- 手動補充備註
- 查看退款狀態

### C. 退款管理
用途：人工處理退款與客訴。

功能：
- 退款申請列表
- 核准 / 拒絕 / 處理中 / 已完成
- 記錄退款金額
- 記錄 admin note

### D. 場次管理
用途：人工處理場次與容量問題。

功能：
- 查看所有活動場次
- 調整容量
- 關閉 / 重開場次
- 顯示 booked_count / capacity
- 查看是否額滿

### E. 營運追蹤（新增，MVP 必做）
用途：追蹤每筆訂單是否健康、是否賺錢、是否需要調整營運策略。

---

## 3. 營運追蹤模組（新增）

### 3.1 模組目的
這個模組不是財務報表，而是 **CEO / 營運決策工具**。

要回答的問題：
- 目前 15% 抽成夠不夠
- 哪些訂單最容易出問題
- 哪些訂單人工成本最高
- 哪些流程最值得優先自動化
- Andy Lee（李衍錫） 模型是否能複製

### 3.2 頁面位置
建議新增：
- `/admin/operations-tracking`
- `/admin/operations-tracking/[orderId]`

### 3.3 列表頁需求
每列代表一筆訂單，需顯示：
- 訂單日期
- 訂單編號
- 導遊名稱
- 活動名稱
- 場次日期
- 旅客人數
- GMV
- 平台抽成收入
- 金流費
- 人工處理時間
- 人工成本估算
- 退款金額
- 是否有例外事件
- 單筆最終貢獻
- 健康訂單標記（是 / 否）

### 3.4 詳情頁需求
Admin 可查看 / 編輯以下欄位：

#### 基本資訊（唯讀或系統帶入）
- 訂單日期
- 訂單編號
- 導遊名稱
- 活動名稱
- 場次日期
- 旅客人數
- GMV
- 平台抽成收入
- 金流費

#### Admin 手動輸入欄位
- 人工處理時間（分鐘）
- 人工成本估算
- 退款金額
- 補貼 / 補償金額
- 是否改期
- 是否退款
- 是否客訴
- 是否導遊臨時調整
- 是否超賣 / 場次異常
- 備註

#### 系統或計算欄位
- 單筆最終貢獻
- 是否為健康訂單

### 3.5 列表篩選條件
- 日期區間
- 導遊
- 活動
- 是否退款
- 是否有例外事件
- 是否健康訂單

### 3.6 匯出需求
MVP 階段支援：
- CSV 匯出

### 3.7 Dashboard Summary（MVP 簡版）
營運追蹤頁頂部顯示：
- 總 GMV
- 平台總收入
- 平均每單抽成收入
- 平均每單人工時間
- 平均每單人工成本
- 退款率
- 例外事件率
- 平均單筆最終貢獻

---

## 4. 操作流程

### 4.1 訂單成立後
- 系統建立營運追蹤紀錄初稿
- 自動帶入可計算欄位

### 4.2 Admin 每週回顧
- 補完人工處理時間與備註
- 標記例外事件
- 檢查健康訂單比例
- 匯出 CSV 供 CEO / 營運檢視

### 4.3 退款或異常後
- 更新退款金額
- 補上事件備註
- 重新計算單筆最終貢獻

---

## 5. MVP 不做的後台功能

- 多層角色權限
- 完整公告系統
- 完整會計報表
- 自動風控系統
- 自動客訴工單流
- 自動 BI 圖表平台

---

## 6. 核心原則

**Admin 後台的任務不是好看，而是能讓平台撐得住、看得懂、救得回來。**
