# Tour Platform MVP Business Rules

> 這份文件的目的只有一個：避免工程、設計、營運各自腦補。
> 更新日期：2026-03-27

---

## 1. 平台模式

### 1.1 MVP 定位
MVP 採 **平台代收款 + 導遊提供服務** 的模式。

也就是：
- 平台展示行程
- 旅客在平台下單與付款
- 平台代收款
- 導遊履約
- 平台依規則結算給導遊

### 1.2 法規注意
此模式可能接近旅行業灰區，因此 MVP 階段：
- 僅聚焦台灣在地導覽 / 體驗
- 不包裝交通、住宿、保險等套裝商品
- 平台條款需明示平台角色與責任邊界
- 正式商轉前需做主管機關諮詢與法律確認

---

## 2. 抽成與結算

### 2.1 平台抽成
- MVP 統一抽成：**15%**
- 不做精選導遊差異抽成
- 不做導遊訂閱制

### 2.2 結算規則
- 訂單完成後，平台才計入可結算收入
- MVP 先採 **人工結算**
- 預設結算週期：**活動完成後 T+7 天**

### 2.3 導遊收益
- 訂單總額 = 旅客付款總額
- 平台抽成 = 訂單總額 × 15%
- 導遊可得 = 訂單總額 × 85%

---

## 3. 產品與定價規則

### 3.1 行程單位
MVP 先統一採 **每人計費（per person）**。

不支援：
- per group 團費模式
- 複雜票種
- 成人 / 兒童 / 長者多層票價

### 3.2 人數規則
- 每個活動有 `min_participants` 與 `max_participants`
- 預約時不可超過該場次可售名額
- MVP 不做候補名單

### 3.3 日期與場次
- 每個活動可有多個可預約日期
- 每個日期視為一個獨立可售場次
- 每個場次有可售名額與狀態

---

## 4. 訂單流程規則

### 4.1 訂單建立
旅客流程：
1. 選活動
2. 選日期
3. 選人數
4. 填聯絡資料
5. 付款
6. 建立訂單

### 4.2 訂單狀態
MVP 訂單狀態如下：
- `pending_payment`：建立中，尚未完成付款
- `paid`：付款成功，待導遊確認
- `confirmed`：導遊接受訂單
- `rejected`：導遊拒絕訂單
- `cancelled_by_user`：旅客取消
- `cancelled_by_guide`：導遊取消
- `completed`：活動完成
- `refund_pending`：退款申請中
- `refunded`：已退款

### 4.3 場次與名額控制機制
MVP 採 **導遊先建立可預約日期／場次，旅客只能預約已開放場次** 的模式。

規則如下：
- 導遊必須先建立活動可預約日期與名額
- 旅客在前台只能看到該活動已開放的日期
- 例如：Andy 的「柴山探洞」若只開放 4/1、4/3，旅客就只能預約 4/1 或 4/3
- 每個場次都必須有 `capacity` 與目前 `booked_count`
- 付款成功後，系統即時增加已預約人數
- 若 `booked_count >= capacity`，該場次自動改為不可預約或 `full`
- MVP 階段保留 Admin / 導遊人工調整名額與場次狀態的能力

### 4.4 訂單確認與名額占用
MVP 原則上採 **付款成功即視為名額占用成功**。

也就是：
- 旅客選定日期與人數後進入付款
- 付款成功 → 訂單成立，並即時反映到該場次名額
- 場次若滿額，前台需立即停止該日期的進一步預約

MVP 階段若發生例外（例如人工改期、超賣修正、導遊臨時調整），由 Admin 人工介入處理。

### 4.5 人工介入原則
MVP 必須保留人工介入能力，至少包括：
- 手動調整場次名額
- 手動關閉 / 重開某一天的預約
- 手動修正訂單狀態
- 手動處理超賣或改期

### 4.6 導遊回應 SLA
- 導遊需在 **24 小時內** 查看新訂單與特殊需求
- 若發生例外狀況，由 Admin 人工介入
- MVP 不做複雜自動派單或候補機制，先以人工營運補足

## 5. 取消與退款規則

### 5.1 MVP 退款原則
MVP 先採 **平台自訂簡化版退款規則**，但正式商轉前需再與法規版核對：

- 活動 7 天（168 小時）前或更早取消：100%
- 活動 4–7 天（>72 小時且 <168 小時）取消：70%
- 活動 72 小時內取消：0%
- 導遊取消：100% 退款

### 5.2 退款狀態
- `requested`
- `reviewing`
- `approved`
- `processing`
- `refunded`
- `rejected`

### 5.3 退款操作
- 旅客可在訂單頁提出退款申請
- 若符合明確規則，可由 Admin 快速批准
- 若有爭議，客服人工處理
- MVP 退款流程以人工為主

---

## 6. 導遊規則

### 6.1 導遊上架資格
導遊需完成：
- 基本資料
- 身分驗證文件
- 銀行帳戶資訊
- 個人照片
- 自我介紹

才可進入審核。

### 6.2 導遊狀態
- `pending`
- `approved`
- `rejected`
- `suspended`

### 6.3 導遊取消處分
- 第 1 次：記錄並警告
- 第 2 次：人工審核是否暫停接單
- 惡意爽約：可直接 `suspended`

MVP 不做自動分級演算法，由 Admin 人工判定。

---

## 7. 評價規則

### 7.1 誰可以評論
- 只有 `completed` 訂單的旅客可以留言評價
- 每筆訂單只能評價一次

### 7.2 MVP 是否必做評價
評價系統可做 **簡化版**：
- 星等
- 短文字評論
- 顯示為已驗證訂單

不做：
- 導遊回覆評論
- 圖片評論
- 檢舉流程自動化

---

## 8. 通知規則

### 8.1 MVP 通知通道
- 只做 Email
- 不做 LINE Notify / 簡訊 / push

### 8.2 需通知事件
- 訂單建立
- 付款成功
- 導遊接受訂單
- 導遊拒絕訂單
- 訂單取消
- 退款狀態更新
- 活動前提醒（可人工）

---

## 9. Admin 規則

### 9.1 Admin 最小能力
Admin 必須能：
- 審核導遊
- 查看全部訂單
- 修改退款狀態
- 標記導遊狀態
- 查看 KYC 文件

### 9.2 MVP 不做的 admin 能力
- 精細 RBAC
- 完整報表系統
- 自動風控
- 自動化客訴工作流

---

## 10. 權限規則

### 10.1 旅客
可：
- 瀏覽活動
- 下單
- 付款
- 查看自己的訂單
- 申請退款
- 對已完成訂單評價

不可：
- 修改導遊資料
- 查看他人訂單

### 10.2 導遊
可：
- 編輯自己的個人資料
- 建立 / 編輯自己的活動
- 管理自己的場次
- 查看自己的訂單
- 接受 / 拒絕自己的訂單

不可：
- 查看其他導遊資料後台
- 手動改付款狀態
- 越權退款

### 10.3 Admin
可：
- 管理所有導遊
- 查看所有訂單
- 更新退款與導遊審核狀態

---

## 11. MVP 明確不支援事項

- 多語 UI
- 多幣別
- LINE Pay
- 複雜票種
- 自動分潤
- 導遊訂閱制
- 替代活動推薦
- 站內聊天
- 自動客服工單
- 動態抽成
- 完整緊急救援調度

---

## 12. 最後原則

任何需求如果不能直接提升以下四件事之一，優先不做：
- 導遊上架
- 旅客下單
- 付款成功
- 履約完成
