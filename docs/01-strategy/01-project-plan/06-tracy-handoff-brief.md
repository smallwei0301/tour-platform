# Tracy Handoff Brief — Tour Platform MVP

> 這份文件是給 Tracy 的正式交接摘要。目標：快速理解、不要重做決策。
> 更新日期：2026-03-27

---

## 1. 專案目標

做出一個 **可交易的台灣在地導遊平台 MVP**，先驗證：
- 導遊願不願意上架與管理場次
- 旅客願不願意為在地導遊行程付款
- 平台能不能用低自動化 + 人工營運方式撐起第一批訂單

這不是內容網站，也不是完整 OTA。
這是一個 **先求成交、履約、可營運** 的 MVP。

---

## 2. Tracy 這一版要做什麼

### 最核心的 4 件事
1. 導遊可以申請加入平台
2. 導遊可以建立活動與可預約日期
3. 旅客可以選擇導遊與日期並完成付款
4. Admin 可以人工介入處理例外、退款與營運問題

---

## 3. 產品原則

### 3.1 交易優先
任何功能若不能幫助以下四件事，優先序就下降：
- 導遊上架
- 場次管理
- 旅客付款
- 訂單履約

### 3.2 允許人工營運
MVP 不是全自動系統。設計時要刻意保留：
- Admin 手動修正狀態
- Admin / 導遊手動調整場次容量
- 手動退款處理
- 手動補發通知（站外可人工）
- 訂單異常人工備註

### 3.3 狀態必須清楚
頁面與資料模型都要能看出：
- 活動是否可訂
- 哪天可訂
- 已售幾人 / 可售幾人
- 訂單現在在哪一個狀態
- 退款處理到哪一步

---

## 4. 已拍板決策（不要重問）

- 平台模式：平台代收款
- 抽成：固定 15%
- 定價：per person
- 場次：導遊先建立可預約日期與容量
- 名額：付款成功後即時占位
- 場次滿額：自動暫停預約
- 通知：先做 Email
- 退款：人工主導
- 後台：最小營運能力
- 評價：簡化版
- 要有 audit trail

---

## 5. 關鍵使用者流程

### Flow A：導遊加入平台
- 申請
- 上傳 KYC 文件
- Admin 審核
- 通過後可登入導遊後台

### Flow B：導遊建立供給
- 建活動
- 建日期 / 場次
- 設定每個場次的人數上限
- 前台顯示可預約日期

### Flow C：旅客下單付款
- 看活動
- 選導遊
- 選日期
- 付款
- 付款成功後占位
- 滿額即停售該日期

### Flow D：營運介入
- Admin 可查看訂單
- 可查看退款
- 可人工修正狀態
- 可人工關閉 / 重開場次

---

## 6. Tracy 特別要注意的 UX 細節

### 日期不是日曆裝飾，是供給核心
- 旅客不能任選日期
- 只能選導遊已開放的日期
- 前台要清楚顯示哪些日期可預約、哪些額滿、哪些不可選

### 場次名額要即時反映
- 付款成功後，名額要立即變動
- 若滿額，要立即阻止後續預約
- MVP 允許後台人工修正，但前台邏輯不能模糊

### 管理後台不用華麗，但要能救火
後台至少要讓營運方可以：
- 找到訂單
- 看出問題
- 手動介入修正

### P1

- Email 模板整理
- 基本 dashboard summary

### P2

- 後台體驗優化
- 更多內容頁
