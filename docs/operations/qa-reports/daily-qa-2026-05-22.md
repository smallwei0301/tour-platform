### #654 ECPay ON CONFLICT fix

**File:** `apps/web/app/api/payments/ecpay/create/route.ts`
**File:** `apps/web/src/lib/ecpay-create-orchestration.mjs`
**File:** `apps/web/src/lib/db.mjs` (`upsertEcpayPaymentAttemptDb`)

Findings:
- The route calls `upsertEcpayPaymentAttemptDb` which first queries for an existing `pending` payment attempt for the same `(order_id, provider='ecpay')` before inserting. If one exists, it returns the existing `merchantTradeNo` (reuse path). If not, it inserts a new row.
- Insert payload contains: `order_id`, `provider`, `merchant_trade_no`, `amount_twd`, `currency`, `status`, `provider_status`, `updated_at`. No `method` column is written.
- No raw SQL `ON CONFLICT` clause is used; the upsert is implemented via a select-then-insert pattern, which is functionally equivalent and avoids column-name issues.
- `buildEcpayCheckoutParams` in `ecpay-create-orchestration.mjs` constructs the ECPay params purely from input; no DB writes occur there.

**Verdict: PASS** — No legacy `method` column write; idempotent reuse of existing pending attempt is implemented correctly.

### #658 Payout date source

**File:** `apps/web/app/api/guide/payout/monthly/route.ts`

Findings:
- Route batch-fetches `activity_schedules.start_at` for all `schedule_id` values found in the month's orders.
- Per-order `scheduleDate` is set to `start_at.slice(0, 10)` from the `activity_schedules` table — this is the tour date, not `created_at`.

- Contract tests (issue655 8/8 pass) confirmed: CSV route uses `scheduleDates` lookup for schedule date, and falls back to `needs_manual_review` string (not `created_at`).

**Verdict: PASS** — Month bucketing uses `activity_schedules.start_at` for tour date; no `created_at`-based date exposed to the guide.
