## 一、我們在做什麼（30 秒版本）

**Tour Platform** 是一個台灣在地導遊直接銷售體驗行程的交易平台。

旅客不再需要透過 KKday、Klook 等大型平台找行程，而是可以直接找到會帶你去柴山洞穴的那位真實導遊、選好日期、當場付款，拿到電子憑證。導遊不需要支付高達 25–35% 的平台抽成，我們只收 **15%**，且導遊有完整的自助後台管理場次與訂單。

**Beachhead 市場：** 高雄柴山戶外探洞體驗，第一位導遊 Andy Lee（李衍錫）。

## 二、從 MVP 到今天：完整開發里程碑

### 🟢 階段一：前台 MVP

打造旅客完整購買旅程：
- 首頁、行程列表、行程詳情（KKday 風格雙欄 layout）
- 導遊個人頁、申請導遊頁
- **3-step 預訂流程**（選場次 → 填資料 → 付款）
- 訂單查看、退款申請
- 法律頁面（服務條款、隱私政策、退款規則）
- 完整行動版（RWD）

**成果：** 旅客可以從首頁到付款完成整個購買旅程（含模擬付款）。

---

### 🟢 階段二：Admin 後台核心

打造平台運營人員的管理工具：
- **訂單管理**：列表、詳情、備註、退款審核、審計日誌
- **導遊審核**：申請列表 + 核准 / 退件
- **營運追蹤**：每筆訂單的 contribution margin、人工成本、健康度指標、CSV 匯出
- **KPI 設定**：可配置計算器（抽成比例、付款手續費），支援版本控制與一鍵 rollback
- **Admin 安全**：email + token 雙重驗證、session 過期、Token 輪替、強制登出

**成果：** 營運人員可以在後台完整管理訂單、退款與財務數據，不需要直接操作資料庫。

---

### 🟢 階段三：UI 精修

對齊 KKday 的設計語言，讓旅客有熟悉的視覺體驗：
- 全站 UI 重設計（KKday 風格色系、字體、元件）
- 行動版 layout 修正（漢堡選單、overflow 問題）
- Andy Lee 資料整合（真實導遊資料、行程照片）
- 日期選擇器優化（30 天捲軸、月曆 modal）

### 🟢 階段四：行程後台 + 付款閉環（2026 Q1 末 — 2026-04-01）

**這是本次報告最重要的階段，也是讓平台從「展示原型」變成「真實可用產品」的關鍵步驟。**

#### 4a. Supabase DB 正式接線（2026-03-31）
- 前台行程資料從 hardcode fixtures 切換到 Supabase PostgreSQL DB
- 3 位導遊、3 個行程、8 個場次、9 筆旅客評價全部入庫

#### 4b. Admin 行程管理（2026-03-31）
- Admin 可在後台新增、編輯、發佈、下架、刪除行程
- 所有欄位可後台編輯：標題、描述、價格、地區、圖片、FAQ、行程時間軸、旅客評價語錄等

#### 4c. 方案管理系統（2026-04-01）
為了符合導遊實際銷售邏輯（一個行程可以有「半日方案」和「全日方案」）：
- **方案管理（PlanEditor）**：每個行程可設定多個方案，各方案有獨立定價、語言、行程介紹（含圖片）、集合地點、退款政策
- **方案詳情 Modal（7 個 tab）**：旅客可在選方案時查看完整的費用包含、行程介紹、集合地點、購買須知與取消政策
- **方案 × 日期排程**：Admin 可開啟特定日期的特定方案，設定容量，前台即時顯示剩餘名額

#### 4d. 圖片管理（2026-04-01）
- 圖片上傳至 **Supabase Storage**（免費、CDN backed）
- 瀏覽器端 WebP 壓縮（最大 1200px），零 server overhead
- 手機端圖片 **Carousel 左右 swipe**，桌機端 3:1 grid layout

#### 4e. JSON 行程匯入/匯出（2026-04-01）
- Admin 可下載「柴山秘境之旅」豐富版 JSON 樣板
- 一鍵匯入 JSON → 欄位驗證 → diff 預覽 → 儲存
- 大幅降低填寫一個完整行程所需時間（從 2 小時 → 5 分鐘）

#### 4f. 付款 → 訂位扣量完整閉環（2026-04-01）✅ 剛完成

**這是最重要的一個里程碑。**

| 之前（有問題） | 現在（已修復） |
|---|---|
| 付款成功後 `booked_count` 用 SELECT + UPDATE（非原子），兩人同時付款可能超賣 | 改用 PostgreSQL RPC `fn_book_schedule`（`SELECT FOR UPDATE` 鎖），完全原子 |
| Booking 頁面讀 hardcode fixtures，價格與 DB 可能不一致 | Booking 頁面讀 `/api/activities/[slug]`，確保價格與 DB 一致 |
| 從行程頁點「立即預約」，booking 頁需重選場次 | URL query params 帶 scheduleId/planId/date，到 booking 頁自動預選 |
| 付款後行程頁剩餘名額要等到下次重啟才更新 | ISR 60 秒 + `router.refresh()`，付款後 60 秒內即時反映 |

## 三、現在的產品能做什麼？

**完整的端對端流程（現在就能跑）：**

```
旅客 → 行程詳情頁
       → 選日期（月曆）
         → 選方案（半日 / 全日）
           → 看方案詳情（7 tabs）
             → 點「立即預約」
               → Booking 頁（自動帶入場次）
                 → 填聯絡資訊
                   → 付款（模擬 ECPay）
                     → 名額即時扣減（原子 RPC）
                       → 訂單成功頁
```

**Admin 後台（現在就能操作）：**
```
Admin 登入
  → 新增行程（或 JSON 匯入）
    → 設定方案（A. 半日 / B. 全日）
      → 上傳圖片
        → 設定排程（開放哪些日期 × 方案 × 容量）
          → 發佈
            → 前台即時出現，旅客可預約
```

## 四、技術架構（給技術 / 投資人）

| 層 | 技術 | 說明 |
|----|------|------|
| 前端 | Next.js 15（App Router） | SSR + ISR + Client Components |
| DB | Supabase PostgreSQL | 13 張表，含 RPC 函數與 trigger |
| 圖片儲存 | Supabase Storage | activity-images bucket，CDN backed |
| 部署 | Vercel | Preview + Production，CI/CD 自動 |
| 付款 | ECPay（模擬） | 真實串接是下一 Sprint |
| 原子扣位 | PostgreSQL RPC `fn_book_schedule` | `SELECT FOR UPDATE`，防超賣 |

**DB 規模（截至 2026-04-01）：** 13 張表、6 次 migration、2 個 RPC 函數、1 個 trigger
