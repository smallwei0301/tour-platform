# 導遊入駐 SOP

> 最後更新：2026-05-26
> 狀態：Draft v2（補齊 KYC 保存規則、通知範本、QA 查核表）
> Owner：Wei
> 關聯：#606 (KYC data-minimization), #601 (payout schema), evidence-artifact-governance.md

## 1. 目的
建立導遊從申請、審核、上架到可接單的標準流程，降低人工 onboarding 混亂。

## 2. Onboarding 流程
1. 導遊提交申請
2. 平台進行基本資格審核
3. 補件 / 退件 / 通過
4. 建立導遊後台帳號或邀請連結
5. 協助完成第一個活動上架
6. 確認付款 / 結算 / 客服規則已理解
7. 正式開放接單

## 3. 建議必填資料
- 姓名 / 品牌名稱
- 聯絡方式
- 可服務地區
- 專長主題
- 導覽經驗
- 身分驗證 / 證照（若適用）
- 收款帳戶資訊（可後補）

## 4. 審核結果分類
### 通過
- 基本資料齊全
- 主題與平台定位相符
- 能提供可售體驗內容

### 補件
- 缺基本介紹 / 行程內容 / 圖片
- 證照或身份資料不完整

### 退件
- 主題與平台定位不符
- 服務內容有法規風險
- 無法證明基本可信度

## 5. 通過後的第一週目標
- 完成導遊帳號啟用
- 完成第一個活動資料建立
- 補齊封面 / 圖片 / 行程細節
- 確認退款 / 結算 / 接單規則理解

## 6. 與文件的連動
- 導遊需閱讀 dashboard 操作手冊
- 導遊需理解退款政策與結算規則
- 導遊需知道客服 / 緊急事件回報窗口

## 7. KYC 文件保存規則

> 對齊：`docs/security/evidence-artifact-governance.md` § 1–2、`docs/05-business/06-payment-plan/05-settlement-payout-ops-runbook.md` § 5

### 7.1 Onboarding 可收集的資料（必要最小化）

| 資料項目 | 必要性 | 說明 |
|---------|-------|------|
| 姓名 / 品牌名稱 | 必填 | 本名驗證用 |
| 聯絡方式（電話 / Email） | 必填 | 審核與通知用 |
| 身分驗證文件（身分證正反面） | 必填 | 實名驗證，資料加密保存於受控位置 |
| 導遊執照 / 相關證照 | 若適用 | 「已持照」徽章顯示條件 |
| 收款帳戶資訊（銀行帳號、戶名） | 必填（可後補） | 結算轉帳用，僅由 Maker/Checker 存取 |

### 7.2 禁止進入 repo / GitHub / Issues / CI 日誌的資料

以下資料屬於 **Sensitive（不得保存）** 或 **Restricted（受控）** 等級（依 `evidence-artifact-governance.md` §1），**絕對不得**出現在：
- GitHub repo（任何分支、任何 commit）
- GitHub Issues / PR 評論
- CI 日誌（GitHub Actions、Vercel 建置輸出）
- 任何 PR 的程式碼變更或 diff

| 禁止進入公開位置的資料 | 原因 |
|----------------------|------|
| 真實身分證字號 | 直接個資 |
| 真實銀行帳號 / 戶名 | 金融個資 |
| 帶識別標記的真實照片（含 EXIF） | 個資 / 隱私 |
| 真實 Email 地址（完整） | 直接個資 |
| 真實電話號碼 | 直接個資 |
| KYC 文件影本（原檔） | Sensitive 等級 |

違規處理：立即啟動 `docs/security/evidence-artifact-governance.md` §7 外洩升級路徑（通知 Wei → P0 事件 #529）。

### 7.3 KYC 資料存放位置與存取控制

| 項目 | 規則 |
|------|------|
| **存放位置** | 受控位置（`/root/.openclaw/workspace/` 或 Wei 指定的加密存儲），**不進 repo** |
| **Owner** | Wei（主要保管人）|
| **存取權限** | 僅限 Wei 及 Maker/Checker 流程中的第二位確認人（有限授權）|
| **傳輸方式** | 加密通道（不得使用 GitHub Issues、Slack 公開頻道傳輸原始文件）|

### 7.4 資料保留與刪除週期

對齊 `05-settlement-payout-ops-runbook.md` §5：

| 階段 | 時間 | 動作 |
|------|------|------|
| **使用中** | 0–90 天 | 保存於受控位置，僅限必要存取 |
| **封存** | 第 90 天 | 封存（移入冷儲存 / 加密壓縮），記錄封存日期與操作人 |
| **刪除評估** | 第 1 年 | Wei 評估是否需繼續保留（例如爭議仍未結） |
| **刪除** | 評估通過後 | 安全刪除原始文件，記錄刪除時間至 audit log |

### 7.5 補件 / 退件 / 刪除的稽核軌跡

每次以下操作須於受控位置留存一筆稽核紀錄（以下為格式，使用佔位符）：

```
[YYYY-MM-DD] [操作類型: 收件/補件/退件/封存/刪除]
導遊 ID：[guide_id 前 8 碼]-[REDACTED]
操作人：[角色名稱]
說明：[操作原因摘要，不得含真實個資]
```

## 8. 通知範本

> 所有範本僅使用佔位符（`[導遊姓名]`、`[活動名稱]`、`[日期]` 等）。
> **嚴禁**在範本中填入真實姓名、身分證字號、銀行帳號、密碼、Token 或任何真實個資。

### 8.1 歡迎信（帳號已建立）

```
主旨：歡迎加入 祕島（Midao）— 您的導遊帳號已建立！

[導遊姓名] 您好，

感謝您申請加入祕島（Midao）導遊平台！您的帳號已通過審核並成功建立。

接下來請完成以下步驟：

1. 登入導遊後台：[後台網址]
2. 完善您的個人頁面（頭像、自我介紹、專長地區）
3. 建立您的第一個活動（預計完成日期：[日期]）
4. 確認收款帳戶資訊已補齊（如尚未填寫）
5. 閱讀退款政策與結算規則

若有任何問題，請聯繫客服窗口：[客服聯絡方式]
```

## 9. Onboarding QA 查核表

> 每次 onboarding 流程結束（導遊帳號啟用）後，執行人須確認以下所有項目。
> 查核結果記錄於受控位置，遮蔽摘要可進 GitHub issue comment。

### 9.1 通知範本 PII 查核

| 查核項目 | 確認 |
|---------|------|
| 所有通知範本僅使用佔位符（如 `[導遊姓名]`、`[日期]`）| ☐ |
| 範本中無真實姓名 | ☐ |
| 範本中無身分證字號 | ☐ |
| 範本中無銀行帳號 / 戶名 | ☐ |
| 範本中無密碼 / Token / API Key | ☐ |
| 範本中無真實 Email 或電話號碼 | ☐ |

### 9.2 KYC / 個資不進 Repo 查核

| 查核項目 | 確認 |
|---------|------|
| 本次 onboarding 未將導遊真實姓名寫入 repo、issues 或 PR | ☐ |
| 本次 onboarding 未將身分證字號寫入任何公開位置 | ☐ |
| 本次 onboarding 未將銀行帳號 / 戶名寫入任何公開位置 | ☐ |
| 本次 onboarding 未將 KYC 文件影本上傳至 GitHub | ☐ |
| CI 日誌中無個資（已在 CI 環境變數中避免輸出） | ☐ |

### 9.3 資料保留規則確認

| 查核項目 | 確認 |
|---------|------|
| KYC 文件已存放於受控位置（非 repo） | ☐ |
| Owner（Wei）已確認收到並可存取 | ☐ |
| 封存日期（第 90 天）已記錄於 audit log | ☐ |
| 刪除評估日期（第 1 年）已記錄於 audit log | ☐ |

### 9.4 與 #601 Payout Schema 一致性確認

| 查核項目 | 確認 |
|---------|------|
| 導遊收款帳戶欄位（銀行帳號、戶名）與 #601 schema contract 一致 | ☐ |
| `guide_balances` 中無明文個資（僅含 guide_id、金額） | ☐ |
| `payouts` 表欄位符合 `05-settlement-payout-ops-runbook.md` §4 規範 | ☐ |
| Payout 對帳 evidence 中銀行帳號已遮蔽（僅存於受控位置） | ☐ |
