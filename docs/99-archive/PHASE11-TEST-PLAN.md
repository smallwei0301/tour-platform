# Phase 11 Task #1 — 手動測試計畫

**文件版本**: v1.0  
**建立日期**: 2026-04-08  
**執行預期**: 20-30 分鐘  
**測試模型**: anthropic/claude-haiku-4-5  
**測試工具**: Playwright (E2E)  

---

## 測試目標

驗證照片上傳功能在以下場景中正常運作：
1. 導遊後台頭像上傳（正常 + 邊界情況）
2. 行程編輯 Hero 上傳（正常 + 邊界情況）
3. 行程編輯 Gallery 上傳（多圖 + 邊界情況）
4. 前台 Placeholder 顯示（無圖時的 fallback）

---

## 測試環境準備

### 預置條件

```bash
# 確保本地環境已有
✅ tour-platform 代碼最新（main 分支，commit 4f74a9b）
✅ Supabase Storage buckets 存在
   - guides (5MB)
   - activity-images (10MB)
✅ 前台環境變數配置正確
   - NEXT_PUBLIC_SUPABASE_URL
   - NEXT_PUBLIC_SUPABASE_ANON_KEY
```

### 測試帳戶

```
Admin 帳戶（用於 /admin/guides 和 /admin/activities）
  ├─ Email: 待定（需從本地取得）
  └─ 權限：能訪問 admin 頁面

指南開發伺服器
  ├─ 啟動：npm run dev
  ├─ URL: http://localhost:3000
  └─ 確認無 TypeScript 錯誤
```

## 測試用例

### TC-1：導遊頭像上傳（正常流程）

**優先級**: P0（阻礙上線）  
**預期時間**: 5 分鐘

**前置條件**
- [ ] 已進入 /admin/guides
- [ ] 頁面切換到「Profiles」標籤
- [ ] 至少有 1 個已審核導遊
- [ ] 點擊某導遊的「編輯」按鈕打開編輯對話框

**測試步驟**

| # | 動作 | 預期結果 |
|---|------|---------|
| 1 | 點擊頭像圓形區域（顯示 👤 icon） | 文件選擇對話框開啟 |
| 2 | 選擇本地 500×500 px JPG 檔（1.5MB） | 無上傳即時開始 |
| 3 | 觀察進度訊息 | 顯示「裁切壓縮中⋯」→ 「上傳中⋯ (XYZ KB)」 |
| 4 | 等待上傳完成 | 頭像預覽更新，顯示已上傳圖片 |
| 5 | 檢查成功訊息 | 頁面顯示「✅ 頭像已更新」 |
| 6 | 重新整理頁面 | 頭像仍顯示已上傳圖片（URL 已存 DB） |
| 7 | 檢查 Supabase | Storage → guides bucket → 確認檔案存在<br/>路徑應為：`{guideId}/avatar-{timestamp}.webp` |

**驗收條件**
- [x] 上傳成功，進度訊息正確
- [x] 圖片自動壓縮為 WebP 400×400
- [x] 預覽更新，URL 存入 guide_profiles.profile_photo_url
- [x] 重新整理後仍顯示已上傳頭像

**備註**
- 自動壓縮：任何尺寸輸入 → 中心裁切 + 縮放到 400×400
- 檔名格式：`avatar-{timestamp}.{ext}` (WebP 優先)

## 測試執行步驟

### 1️⃣ 環境準備（5 分鐘）

```bash
# 確保本地開發伺服器運行
cd /root/tour-platform/apps/web
npm run dev

# 等待 "ready - started server on 0.0.0.0:3000"
# 檢查無 TypeScript / 編譯錯誤
```

### 2️⃣ Playwright 自動化測試（使用 Haiku 模型）

```bash
# 執行 E2E 測試
npx playwright test tests/phase11-photo-upload.spec.ts --project chromium

# 或使用 Haiku 模型編寫和執行測試腳本
# 見下方「使用 Haiku 執行」
```

### 3️⃣ 手動驗收測試（如果自動化測試不完整）

- 打開 Chrome → http://localhost:3000/admin/guides
- 依序執行 TC-1 到 TC-8

### 4️⃣ 記錄結果

- 每個 TC 記錄 PASS / FAIL
- FAIL 時記錄錯誤訊息和重現步驟
- 截圖（如需要）
